# 20 項 UI/UX 審查原則與程式碼層檢核點

每項原則包含:定義、在程式碼中的具體檢核點、常見違反樣態。
審查時逐項對照,給出 ✅/⚠️/❌/➖/🔍 狀態。

---

## 第一組:Nielsen 十大易用性啟發法(P1–P10)

### P1 系統狀態可見性(Visibility of System Status)
使用者隨時知道系統正在做什麼。
**檢核點:**
- 非同步操作(fetch、表單送出、檔案上傳)是否有 loading 指示(spinner/skeleton/進度條)
- 按鈕送出後是否 disabled,防止重複提交
- 操作成功/失敗是否有明確回饋(toast、訊息區),而非靜默完成
- 多步驟流程(如分級測驗)是否顯示目前進度(第 X 題/共 Y 題)
**常見違反:** `fetch()` 後直接更新畫面,無 loading 與 error 狀態;送出按鈕可連點。

### P2 系統與真實世界的對應(Match Between System and Real World)
用使用者的語言,不用工程術語。
**檢核點:**
- 錯誤訊息是否人話(「找不到此課程」vs「Error 404: resource not found」)
- 介面文字是否符合目標族群慣用語(華語學習者 vs 教師後台,用語深度不同)
- 圖示是否符合通用慣例(垃圾桶=刪除、放大鏡=搜尋)
- 日期、數字格式是否符合在地慣例

### P3 使用者控制與自由(User Control and Freedom)
提供「逃生出口」。
**檢核點:**
- 多步驟流程能否返回上一步而不遺失已填資料
- 破壞性操作(刪除、清除進度)是否可取消或有確認對話框
- Modal/燈箱是否可用 Esc 或點擊外部關閉
- 瀏覽器「上一頁」行為是否符合預期(SPA 路由是否正確管理 history)

### P4 一致性與標準(Consistency and Standards)
同樣的事在整個系統長一樣。
**檢核點:**
- 同類按鈕(主要/次要/危險)樣式是否全站一致
- 同一動作的措辭是否一致(不要一頁叫「送出」、另一頁叫「確認」、又一頁叫「提交」)
- 元件是否抽共用(若每頁複製貼上一份按鈕樣式 → ❌,這也是維護風險)
- 間距、圓角、陰影是否使用統一的 design token / CSS 變數

### P5 錯誤預防(Error Prevention)
讓錯誤難以發生,優於讓錯誤好恢復。
**檢核點:**
- 表單欄位是否有前端驗證與適當的 input type(email、number、date)
- 是否在送出「前」即時提示格式錯誤,而非送出後才報錯
- 危險操作的確認對話框是否說明後果(「將刪除全部 12 筆紀錄」vs「確定嗎?」)
- 是否用選擇取代輸入(下拉/日期選擇器),減少自由輸入錯誤

### P6 辨識優於回憶(Recognition Rather Than Recall)
不要求使用者記住跨畫面的資訊。
**檢核點:**
- 已選擇的設定/篩選條件是否持續顯示在畫面上
- 跨步驟流程,前面填過的關鍵資訊是否在後續步驟可見(如確認頁顯示摘要)
- 輸入框是否有 placeholder 或範例提示格式
- 導覽是否標示目前所在位置(active 狀態)

### P7 彈性與使用效率(Flexibility and Efficiency of Use)
新手能用,熟手能快。
**檢核點:**
- 常用操作的點擊深度(核心任務是否 ≤3 步可達)
- 是否記住使用者上次的選擇(localStorage/偏好設定)
- 表單是否支援 Enter 送出、欄位間 Tab 順序是否合理
- 長列表是否有搜尋/篩選

### P8 美學與極簡設計(Aesthetic and Minimalist Design)
每個畫面元素都該為任務服務。
**檢核點:**
- 單一畫面的主要行動(primary action)是否只有一個、視覺上突出
- 是否有與任務無關的裝飾元素搶佔注意力
- 資訊密度:行動裝置上單屏資訊量是否過載
- 留白是否足夠,區塊分組是否清晰

### P9 協助使用者辨識、診斷並從錯誤中恢復(Error Recovery)
錯誤訊息要說「發生什麼+怎麼辦」。
**檢核點:**
- catch 區塊是否給使用者可行動的訊息(「網路連線失敗,請檢查網路後重試」+ 重試按鈕)
- 表單驗證錯誤是否標示在「該欄位旁」,而非只在頁面頂部
- 404/錯誤頁是否提供返回路徑
- console.error 是否同時有對應的使用者可見回饋(只寫 console = 使用者面前靜默失敗)

### P10 說明與文件(Help and Documentation)
最好不需要,但需要時找得到。
**檢核點:**
- 複雜功能首次使用是否有引導(onboarding/工具提示)
- 專有名詞(如 TBCL 等級)是否有就地說明(tooltip/說明連結)
- 空狀態(empty state)是否引導使用者下一步,而非空白一片

## 第二組:互動心理學定律(P11–P16)

### P11 Fitts's Law(目標大小與距離)
越常用的目標應該越大、越容易點到。
**檢核點:**
- 觸控目標是否 ≥44×44px(iOS 標準)/48×48px(Material 標準),含 padding
- 行動版上相鄰可點元素間距是否 ≥8px,避免誤觸
- 主要行動按鈕是否位於拇指熱區(行動版畫面下半部)
- 小圖示按鈕是否有足夠的可點擊區域(用 padding 擴大,而非只有圖示本身)

### P12 Hick's Law(選項數量與決策時間)
選項越多,決策越慢。
**檢核點:**
- 單一畫面的選項/按鈕數量是否合理(導覽項目建議 ≤7)
- 複雜選擇是否分步驟呈現(漸進揭露),而非一次全攤開
- 設定頁是否分組,而非一長串平鋪

### P13 Jakob's Law(符合既有心智模型)
使用者把在其他網站養成的習慣帶到你的網站。
**檢核點:**
- Logo 是否在左上且點擊回首頁
- 導覽位置是否符合慣例(桌面頂部/行動底部或漢堡選單)
- 連結是否有視覺區別(底線或色彩),按鈕長得像按鈕
- 不要重新發明標準元件的行為(自製下拉選單是否支援鍵盤操作)

### P14 Miller's Law(資訊分塊)
工作記憶約 7±2 個單位,資訊要分塊。
**檢核點:**
- 長表單是否分區塊/分步驟
- 長數字或代碼是否分段顯示
- 內容是否用標題層級組織,而非長段平鋪

### P15 Doherty Threshold(回應時間 <400ms)
系統回應 <400ms 時使用者保持心流。
**檢核點:**
- 互動回饋是否即時(點擊後 UI 立即反應,即使資料還在載入 → optimistic UI 或立即顯示 loading)
- 是否有不必要的阻塞(同步載入大資源、未拆分的大 bundle)
- 圖片是否有尺寸標註(防 layout shift)、是否 lazy loading
- 🔍 實際載入時間需實機/Lighthouse 驗證

### P16 Gestalt 原則(視覺分組)
靠近的、相似的元素被視為同組。
**檢核點:**
- 相關元素(label 與其 input、標題與其內容)的間距是否小於不相關元素
- 卡片/區塊的視覺邊界是否與資訊結構一致
- 對齊:是否有明確的對齊網格,避免元素各自漂移

## 第三組:可及性與適應性(P17–P20)

### P17 可及性基礎(Accessibility / WCAG 2.1 AA)
**檢核點:**
- 文字對比度:一般文字 ≥4.5:1、大字 ≥3:1(依色碼計算理論值,標 🔍 實機確認)
- 所有功能可純鍵盤操作,focus 樣式可見(不要 `outline: none` 而無替代)
- 圖片有 alt、表單欄位有 label(或 aria-label)、按鈕有可讀名稱(純圖示按鈕需 aria-label)
- 語意化 HTML:button 用 `<button>` 而非 `<div onclick>`、標題層級不跳級
- `lang` 屬性正確(`zh-Hant` / `zh-TW`)

### P18 響應式與行動優先(Responsive / Mobile-First)
**檢核點:**
- viewport meta 是否正確設定
- 是否有水平捲動(固定寬度元素溢出)
- 斷點行為:主要斷點下版面是否皆可用(不只「沒壞」而是「好用」)
- 字級在行動裝置是否 ≥16px(避免 iOS 自動縮放),中文行高建議 ≥1.6

### P19 回饋與微互動(Feedback and Microinteractions)
**檢核點:**
- 可互動元素是否有 hover/active/focus 狀態變化
- 狀態轉換是否有適度過渡動畫(150–300ms),而非瞬間跳變
- 動畫是否尊重 `prefers-reduced-motion`
- 觸覺回饋一致性:同類互動的回饋方式全站一致

### P20 Tesler's Law(複雜度守恆)
無法消除的複雜度,應由系統吸收,而非丟給使用者。
**檢核點:**
- 系統能推斷的資訊是否要求使用者填(能自動偵測語言/時區/合理預設值就不要問)
- 預設值是否覆蓋多數使用情境
- 技術細節是否外漏給使用者(URL 參數要求手動修改、要使用者理解內部狀態)
- 容錯輸入:全半形、前後空白、大小寫是否由系統正規化,而非報錯要求使用者重打
