# AI 生成程式碼的常見弱點（優先掃描）

當使用者表明程式碼是用 AI 生成 / vibe coding 寫出來的，先用這份清單做第一輪快掃，再進各技術棧的細部審查。AI 寫的碼通常「能跑」，但安全步驟常被省略或用示範等級的寫法帶過；而且使用者未必逐行讀懂，所以**優先直接修好（✅），並用白話解釋風險**。

## 第一輪快掃順序

1. **殘留的祕密與佔位符** → 2. **注入點** → 3. **認證／授權缺口** → 4. **「為了能跑」而放寬的設定** → 5. **相依套件** → 6. 其他。

## 常見樣態（依出現頻率與危險度）

### 1. 殘留的金鑰、佔位符、測試端點
AI 對話過程中常被貼進真實金鑰，或留下 `YOUR_API_KEY`、demo 帳密、`http://localhost`、測試用 endpoint。
- 全文搜尋 `key`、`secret`、`token`、`password`、`sk-`、`Bearer`、`localhost`、`127.0.0.1`。
- 真實金鑰一律視為已外洩 → ✅ 改成從設定讀取，並提醒使用者 🔧 作廢重發。

### 2. 字串拼接的 SQL / HTML（AI 最愛的「直觀版」）
AI 傾向先寫最直觀、也最不安全的版本：f-string 組 SQL、`innerHTML` 塞變數。
- ✅ 直接改參數化查詢 / `textContent` / 跳脫。

### 3. 只在前端／客戶端驗證
AI 把功能「做出來」時，常只在前端檢查，伺服器端照單全收。
- 標出哪些驗證必須在伺服器端重做；✅ 補上伺服器端驗證。

### 4. 為了「讓它跑起來」而放寬的設定（高危且隱蔽）
這類是 AI 為了排除錯誤最常給的「捷徑」，務必抓出來：
- 關掉 TLS 驗證：`rejectUnauthorized: false`、`verify=False`、`InsecureRequestWarning` 被消音。
- CORS 全開：`Access-Control-Allow-Origin: *`（尤其搭配憑證）。
- `debug=True` / 詳細錯誤頁留在生產環境。
- 過寬權限：Apps Script 部署成 `Anyone`、檔案 `chmod 777`、雲端 bucket 公開。
- ✅ 能在程式碼層收緊的直接收緊；🔧 屬部署／環境的列待辦。

### 5. 缺授權檢查（只顧 happy path）
AI 常只實作「正常流程」，忘了「這個使用者有沒有權限存取這個物件」。
- 找 `GET /resource/:id` 這類有沒有擁有權檢查（IDOR）。✅ 補檢查。

### 6. 錯誤處理把內部細節吐出去
`catch (e) { res.send(e.stack) }`、把例外訊息直接回給使用者。
- ✅ 對外回友善訊息，內部細節只進伺服器 log（且不含 PII）。

### 7. 相依套件：過時、虛構、或拼字變體（AI 特有風險）
AI 可能建議**不存在的套件**或**真實套件的拼字變體**（攻擊者會搶註這些名稱，稱 slopsquatting）。
- 安裝前確認每個套件**確實存在且是你要的那個**（核對官方來源、下載量、維護狀態）。
- 🔧 跑 `npm audit` / `pip-audit` / `osv-scanner` 取得即時漏洞;別憑記憶判斷版本安全與否。

### 8. 複製來、語境不符的片段
AI 拼接的範例碼，其安全假設可能不適用（例如原本假設輸入可信）。
- 檢查每段「看起來很標準」的碼，其信任假設在你的情境下是否成立。

### 9. 缺速率限制 / 資源上限
只做 happy path，沒想到被灌爆。
- 標出對外端點缺 rate limit、清單缺分頁上限（見 stress-and-performance.md）。

### 10. 註解 / console.log 留下敏感資訊
開發過程的除錯輸出留在碼裡。✅ 移除或遮蔽。

## 對使用者溝通的調整

- 假設使用者未必讀懂每一行 → 用白話講「這段在做什麼、為什麼危險」，少用未解釋的術語。
- 預設**直接套用安全修正**（✅），把需要判斷的（⚠️）和需要操作環境的（🔧）清楚分開。
- 誠實的提醒（要講、但別說教）：審查能大幅降低風險，但**不等於讓 AI 生成的碼變成「絕對安全」**。涉及真實個資或對外公開的系統（如諮輔單），上線前仍值得一次人工或專業複核，不要把這份報告當唯一防線。
