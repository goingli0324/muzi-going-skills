# 輸出格式：詳盡風險報告 ＋ 已套用的修正版程式碼

輸出要做到「該修的直接修好、不能替你決定的清楚列出」。發現清單依嚴重度由高到低排序。描述漏洞到「足以理解與修補」即可，不要寫成可直接拿去攻擊的武器化 payload。

## 兩類處理方式（核心區分）

每個發現都要歸到其中一類，並在報告裡用標記標明：

- **✅ 已直接修正**：機械式、低風險、不改變預期功能的修正——輸入驗證、輸出跳脫、公式注入防護、參數化查詢、加 `LockService`、log 遮蔽、把硬寫金鑰改成從設定讀取、加上分頁上限、修正 CORS／安全標頭設定。這些**直接套進第三段的修正版程式碼**，使用者貼上即可用。
- **⚠️ 需你決定**：牽涉產品取捨或會動到使用者體驗的改動——改變回傳格式（HTML→JSON）、改部署的執行身分（影響誰能用）、停用某功能。給出建議與取捨，但保留決定權給使用者。
- **🔧 需你操作**：skill 動不了使用者環境的部分——調整部署設定、作廢並重發金鑰、把祕密值寫進 Script Properties／Secrets Manager、更新相依套件、跑 `npm audit`、做真實壓測、確認個資合規。列成可勾選的待辦。

> 邊界：skill 只在「輸出的程式碼」裡套用修正,**絕不**代替使用者操作其環境(不改部署、不動分享權限、不寫入真實金鑰值、不執行交易)。

## 每個發現要寫多詳細

一筆發現至少包含：類別（含 OWASP 對應）、位置、**問題機制**（為什麼會發生，不只說「有漏洞」）、**攻擊情境示例**（具體但不武器化，讓使用者真的看懂風險）、影響、處理方式標記（✅/⚠️/🔧）、修補（含程式碼）、**驗證方式**（怎麼確認修好了）、取捨、工作量。

# 程式碼安全檢測報告

## 一、總覽
- **受檢範圍**：（檔案／函式／技術棧）
- **執行環境假設**：（是否對外公開、是否處理個資、部署方式——使用者沒講就寫「假設為 ___，若不符請告知」）
- **風險總評**：（最高嚴重度 ＋ 一句話結論）
- **發現統計**：Critical ◯ / High ◯ / Medium ◯ / Low ◯ / Info ◯
- **處理分布**：✅ 已直接修正 ◯ 項 / ⚠️ 需你決定 ◯ 項 / 🔧 需你操作 ◯ 項
- **未能評估的部分**：（靜態審查抓不到的：執行期行為、相依套件即時漏洞、商業邏輯、實際部署設定）

## 二、發現清單（依嚴重度排序）

每筆用此格式：

### [編號] [嚴重度] [✅/⚠️/🔧] 標題
- **類別**：（注入 / 認證 / 授權 / 資料外洩 / 設定 / 效能 / 品質；附 OWASP 對應如 A01）
- **位置**：檔案 ＋ 行號 / 函式名
- **問題機制**：為什麼會發生（具體到程式行為）
- **攻擊情境示例**：一個具體但不武器化的情境，說明攻擊者會怎麼利用、得到什麼
- **影響**：最壞情況
- **修補**：做法 ＋ 修正後程式碼片段（若為 ✅，註明「已套入第三段」）
- **驗證方式**：修完怎麼確認有效（測試輸入、預期行為）
- **取捨／風險**：修了之後要注意什麼
- **工作量**：小 / 中 / 大

## 三、已直接套用的修正版程式碼（✅ 項目）
把所有「✅ 已直接修正」整合成**可直接貼上的完整片段或函式**。用註解標出每處改了什麼、對應哪個發現編號：
```
// [修正#3] 原本 xxx，改為 yyy，防止 zzz
```
改動大時分段呈現並說明套用順序。這段的目標是「貼上就能用」。

## 四、需要你親自處理的（⚠️ ＋ 🔧 項目）
做成可勾選清單，每項寫清楚「要做什麼、為什麼、怎麼做」：
- [ ] 🔧 …（環境／部署／金鑰／合規）
- [ ] ⚠️ …（需決定的取捨，附兩種選項與後果）

## 五、壓力測試與效能
- 風險點清單
- 建議壓測計畫：工具 / 場景 / 指標 / 通過門檻（見 stress-and-performance.md）
- 提醒：計畫非結果；Apps Script 以配額／併發檢查為主

## 六、優化與可調整方向
- 分 **必要**（安全問題）與 **可選**（品質／結構／效能風格）
- 對「可選」項說明採納好處、不採納後果、取捨

## 七、後續建議與限制聲明
- 下一步（呼應第四段待辦）
- 重申：靜態審查 ≠ 滲透測試；報告乾淨不代表系統安全；處理真實個資前不應以此為唯一防線

## 簡版模式
使用者只丟一小段碼、要快速回饋時，可省略第五、六段，保留：總覽（含處理分布與限制）＋ 發現清單 ＋ 已套用修正 ＋ 需你處理清單。「限制」與「需你處理」一定要留。
