# 壓力測試與效能

讀程式碼**無法**替你跑出壓測結果，這點要對使用者講清楚。能做的是兩件事：(a) 靜態找出在高負載下會出問題的「風險點」；(b) 產出一份可執行的「壓測計畫」。

## A. 靜態風險點偵測

### 通用
- **無上限的迴圈／遞迴**：輸入能讓迴圈次數無限放大。
- **缺分頁**：清單型查詢一次拉整張表 → 資料量大時記憶體爆掉、回應變慢。
- **N+1 查詢**：迴圈裡逐筆查資料庫，而非一次批次撈。
- **缺速率限制**：對外端點沒有 rate limit → 易被濫用、爆量、成本失控、DoS。
- **同步阻塞**：大檔同步讀寫、CPU 密集運算卡住事件迴圈（Node）／worker。
- **外部呼叫沒設 timeout**：依賴的服務變慢時，整條請求一起卡死、連線耗盡。
- **記憶體洩漏**：事件監聽器沒移除、快取無上限增長。

### 前端額外
- 大量 DOM 操作造成 reflow/repaint（layout thrashing）。
- 一次渲染上萬個節點而非虛擬化／分頁。

### Apps Script 額外（配額導向）
- 單次執行時間上限（一般帳號約 6 分鐘）；長流程要分批或用觸發器續跑。
- 迴圈內逐格 `getValue()/setValue()` → 改 `getValues()/setValues()` 批次。
- URL Fetch、寄信、觸發器總執行時間都有**每日配額**；高頻呼叫會撞上限。
- 對外部服務呼叫加**指數退避（exponential backoff）**重試。
- 高併發寫同一份 Sheet → 配合 `LockService`（見 apps-script 清單）。

## B. 壓測計畫模板

產出時填好這幾項，讓使用者能直接拿去執行：

**1. 工具（依技術棧選）**
- HTTP API：k6、Artillery、Locust、JMeter
- 前端效能：Lighthouse、WebPageTest（這是效能量測，非負載測試）
- Apps Script：**沒有真正的負載測試工具**。改為「估計尖峰同時使用人數 → 對照配額與執行時間 → 確認有 LockService 與退避機制」，並在非正式部署做小規模並發試跑。

**2. 場景**
- 預期尖峰同時使用人數（concurrent users）
- 升載曲線（ramp-up）：例如 30 秒內從 0 爬到 N
- 持續時間：例如維持尖峰 5 分鐘
- 代表性流程：照真實使用路徑（登入 → 查詢 → 提交）

**3. 量測指標**
- 延遲：p50 / p95 / p99（看尾端，不只看平均）
- 錯誤率
- 吞吐量（RPS / 每分鐘交易數）
- 資源使用（CPU、記憶體、DB 連線數）

**4. 通過門檻（範例，需依實況調整）**
- 在 N 個同時使用者下，p95 延遲 < X ms
- 錯誤率 < 1%
- 無記憶體持續上升（無洩漏跡象）

**5. 必要提醒**
- 在**非正式環境**跑，不要打生產系統。
- 這是計畫；實際數字要跑了才知道。
- 對第三方／雲端服務壓測前，確認沒違反對方的使用條款。
