# 資料外洩專項檢查清單（跨技術棧，一律執行）

這份不分前端／後端／Apps Script 都要走一遍，因為外洩往往不是「漏洞」而是「不小心暴露」。

## 1. 程式碼／版控裡的祕密

- 硬寫的 API key、token、密碼、連線字串、私鑰、`.pem`。
- 檢查 git 歷史——刪掉不代表沒留在 commit 紀錄裡。
- 處置：移到環境變數／祕密管理服務；已外洩的要**輪替（rotate）**，不是只刪掉。

## 2. 送到客戶端的祕密

- 前端 JS、行動 App、minified bundle 裡的任何金鑰都是公開的。
- source map（`.map`）若隨生產環境一起部署，等於把原始碼攤開。

## 3. 個資（PII）處理

- 蒐集最小化：只收真正需要的欄位。
- 傳輸加密（HTTPS）、靜態加密（敏感欄位）。
- 存取控制：限制誰能讀，並留稽核軌跡。
- 保存與刪除政策：不要無限期留存。
- **合規提醒**：在台灣，處理姓名、學號、聯絡方式、諮商紀錄等屬《個人資料保護法》規範範圍。這裡只做提醒，**不是法律意見**；牽涉合規責任時建議諮詢學校個資權責單位。

## 4. 錯誤訊息洩漏

- 對外的錯誤不要帶 stack trace、SQL 錯誤、內部檔案路徑、框架／套件版本。
- 生產環境關閉 debug／verbose 模式。

## 5. Log 寫入敏感資料

- log 通常防護較弱、保存較久、較多人看得到。
- 不要 log 密碼、token、完整個資、信用卡號；需要時遮蔽（如只留末四碼）。

## 6. URL 帶敏感參數

- 把 token、個資放 query string（`?token=...&id=...`）會被記進 server log、瀏覽器歷史，並透過 `Referer` header 洩漏給第三方。
- 敏感資料走 POST body 或 header，不要進 URL。

## 7. CORS / 開放 API

- 過寬的 CORS 讓任意網站讀你的 API 資料。
- 確認端點有適當的存取控制，不是「誰來都給」。

## 8. 暴露的檔案

- web server 對外提供了不該公開的檔案：`.env`、`.git/`、`.bak`、`config.json`、備份檔、資料庫 dump。
- 確認 server 設定有擋掉這些路徑。

## 9. 快取

- 含敏感資料的回應被 CDN／瀏覽器快取，可能被後續使用者讀到。
- 敏感回應加 `Cache-Control: no-store`。

## 10. 中繼資料（Metadata）

- 上傳圖片的 EXIF（含 GPS 位置）、文件的作者／修訂中繼資料可能外洩個資。上傳處理時清除。

## 11. 第三方資料流

- 分析、追蹤、聊天小工具等第三方腳本是否會把表單裡的敏感欄位送出去。
- 嵌入的 iframe / pixel 同樣要檢視。

## 快速自我提問
- 「這份資料如果出現在公開的 server log 裡，會不會出事？」
- 「拿到前端原始碼的人，能看到什麼不該看到的東西？」
- 「這個欄位真的需要回傳／儲存／記錄嗎？」
