# 交棒協定（Handoffs）

本 skill 是編排器，串接三個既有 skill：`project-guardrails`（前置）、`web-security-reviewer`（Verify 安全閘）、`ui-ux-deploy-reviewer`（Verify UI/UX 閘）。交棒不是「把球丟出去就好」，而是把對方需要的脈絡一起交出去，並在對方做完後接回主迴圈。

四個 skill 在迴圈裡的位置：

```
（前置）project-guardrails ──→ 寫入 CLAUDE.md 連動禁區
            │
            ▼
  Research → Plan（讀禁區避雷）→ Build
            │
            ▼
        Verify 雙閘 ┌─ 閘 A：web-security-reviewer（安全/個資）
                    └─ 閘 B：ui-ux-deploy-reviewer（UI/UX，僅當有前端）
            │
            ▼（兩閘皆無未解 P0/Critical/High）
        Deploy（+ 帳號核對）
```

---

## 交棒點 1（前置）→ `project-guardrails`

### 它是什麼
每個專案跑一次的靜態分析工具：掃程式碼找出「改 A 會壞 B」的連動禁區，經使用者確認後寫進該專案 CLAUDE.md。**只分析與寫入，不改程式碼。**

### 何時交
本迴圈第 0 步：專案 CLAUDE.md **沒有**「連動禁區」段落 → 先交棒它建立；架構大改後 → 建議重跑；已有 → 直接讀進來當規劃依據。

### 交什麼 / 怎麼接回來
交棒「這個專案」即可（它自會讀設定檔、grep 統計呼叫點找證據）。本 skill 要確認它分析的是正確 repo，並提醒使用者「禁區草稿要寫進跨專案共用 CLAUDE.md、需逐項確認」。接回來後進規劃，**比對這次改動有沒有踩到禁區**，有就把「驗證下游沒壞」寫進 plan.md 驗收標準。

---

## 交棒點 2（Verify 雙閘）→ 兩個 reviewer

### 先判斷哪些閘要走

| 這次改動的性質 | 走閘 A（安全） | 走閘 B（UI/UX） |
|---|---|---|
| 有前端介面 + 碰個資（多數 my-learning-pwa 功能） | ✅ | ✅ |
| 有前端介面、不碰個資（純展示頁、靜態調整） | 🟡 公開前建議 | ✅ |
| 純後端 / API / 排程（GAS doPost、Cloud Run、無畫面） | ✅ | ➖ 跳過 |
| 🔴 機構系統任何上線 | ✅ 強制 | 有介面才走 |

兩道閘**並行、不互相取代**，且都會讀 CLAUDE.md 連動禁區。

### 閘 A：`web-security-reviewer`（安全/個資/壓力）
交棒時一次給足，讓它能正確定嚴重度：
1. 受檢程式碼（這次 Build 改動的部分）
2. 執行環境（Firebase Hosting / GAS Web App / Cloud Run；對外公開或內部）
3. 是否處理個資（明說「這是 🔴，會存學員姓名/聯絡方式」之類）
4. 部署方式、誰會存取
5. 相依套件情況（有無 package.json/lockfile；純 GAS 線上專案通常不適用相依掃描）

回傳「依嚴重度排序的風險報告 + 修正版程式碼」，分 ✅ 已直接修 / ⚠️ 需你決定 / 🔧 需你操作。

### 閘 B：`ui-ux-deploy-reviewer`（UI/UX + 呈現層）
交棒時給足：
1. 受檢的前端程式碼 / 頁面 / 路由
2. **目標使用者與裝置情境**（例如 my-learning-pwa 是行動裝置上的華語學習者）——它若不明會問，先給可省一輪
3. 關鍵使用者流程（從進入 → 主要任務 → 完成/錯誤路徑）
4. 它會自己讀 CLAUDE.md 連動禁區；確認該檔存在即可

回傳「20 原則問題清單（P0/P1/P2）+ 逐項檢核表 + 部署放行檢核表 + 需實機驗證項目（🔍）」。它的安全議題會主動讓給閘 A，不必擔心重複。

### 接回主迴圈的硬規則
**任一閘有未解的 P0 / Critical / High → 不進 Deploy。** 回到 Build 修掉，必要時更新 plan.md 驗收標準後再驗。⚠️ 與 🔧、以及 🔍（需實機驗證）併入部署待辦，由使用者親自處理或明確接受風險。
修正時一律照全域 CLAUDE.md §二「改動中」(最小 diff、一次一需求)。**特別注意全域 §四 的鐵則:若 reviewer 的修正會讓 smoke test 失敗，只能修測試或回報疑似 bug 等裁決，禁止為了讓測試通過而硬改 `src/`**——那會消滅這道驗證的價值。

### 邊界
兩個 reviewer 都只審查與修補、不替使用者動環境、不寫攻擊程式。本迴圈一致：Verify 是為了安全且好用地上線，不是攻擊測試。

---

## 兩類交棒守同一條線

前置與雙閘傳遞的都是**脈絡與分析結果**，而**決定權與操作（確認禁區、修不修、按不按部署）始終在使用者手上**。這跟三個被串接 skill 各自的硬規定一致：project-guardrails「未確認不寫入」、web-security-reviewer「只審不攻、不動環境」、ui-ux-deploy-reviewer「靜態審查不取代真機/使用者測試，需實機項目交還使用者」。

---

## 附：離開迴圈的轉場（非交棒）
若開發成果要寫成學術產出（研討會摘要、論文章節），屬於**離開本開發迴圈**的轉場，不是本 skill 的交棒關卡。本 skill 不代寫學術文字、不繞過 AI 偵測。
